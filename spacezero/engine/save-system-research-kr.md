# 복셀 엔진 저장 시스템 — 리서치 & 권장사항

**날짜:** 2025-02-25
**맥락:** 창업 엔지니어가 제기한 복셀 엔진 저장/영속성 시스템 관련 8가지 기술적 우려에 대한 응답.

---

## 요약

핵심 긴장 관계 — 편집에 LOD 0 소스 데이터가 필요하지만 메모리 최적화로 제거된 상태 — 는 복셀 엔진에서 잘 알려진 문제입니다. 업계 표준 해결책은: **LOD 0 소스만 디스크에 저장하고, 밉맵은 지연 재생성하며, 더티 플래깅과 디바운스 쓰기를 사용하는 것입니다.** 브라우저 제약(원시 파일시스템 없음)은 이제 OPFS로 대부분 해결됩니다. LOD 레벨 차이 2 이상에서 메싱이 깨지는 문제는 Transvoxel 알고리즘이라는 표준 해법이 있습니다.

가장 어려운 미결 사항은 **월드 크기와 스케일** — 영속성 배포 후에는 되돌릴 수 없기 때문입니다. 이것을 가장 먼저 확정해야 합니다.

---

## 1. LOD 0 청크만 저장해도 되는가?

**답: 네. 이것이 표준 접근법입니다.**

프로덕션 복셀 엔진(Minecraft, Voxel Plugin for UE, Godot Voxel Tools)은 모두 최고 해상도 소스 데이터만 영속화합니다. 하위 LOD 레벨은 파생 데이터이며, 로드 시 또는 필요 시 소스에서 재생성됩니다.

**이유:**
- LOD 1+ 데이터는 LOD 0 소스의 결정론적 함수. 저장하는 것은 중복.
- 밉맵 재생성 비용은 로드 시 한 번만 발생, 지속적으로 발생하지 않음.
- 디스크 용량을 최소로 유지 — 소스 진실만 저장.

**권장:** LOD 0만 저장. 로드 시 밉맵 계층 재생성. 계산 후 밉맵을 메모리에 캐시.

---

## 2. 플레이어 위치 기반 LOD 청크 로딩의 영향

**문제:** 플레이어 근처 청크가 LOD 0이 아니면 편집할 수 없음. LOD 전파 레벨 차이가 2 이상이면 메싱이 깨짐.

**업계 표준: 인접 청크 간 최대 LOD 차이 1을 강제.** 이것은 거의 모든 LOD 복셀 시스템의 하드 제약입니다. Transvoxel 알고리즘(Eric Lengyel, 2009) — LOD 이음새 아티팩트의 표준 해법 — 은 해상도가 정확히 2배(LOD 1레벨) 차이나는 경계를 위해 설계되었습니다. 4배 이상의 차이는 처리하지 못합니다.

**실질적 함의:**
- LOD 전파 로직은 *반드시* 이웃 간 최대 차이 1을 강제해야 함.
- 편집 반경 내 플레이어 근처 청크는 항상 LOD 0으로 강제되어야 함.
- 따라서 플레이어 위치 기반 LOD 로딩은 "있으면 좋은" 기능이 아니라 편집과 올바른 메싱 모두의 전제조건.

**권장:** 플레이어 위치 기반 LOD 로딩 구현을 우선시할 것. 편집 시스템과 메싱 시스템 모두 이에 의존. 플레이어 주변 설정 가능한 편집 반경 내에서 LOD 0 강제.

---

## 3. LOD 1 이상 청크 편집을 허용하지 않는 이유

**답: 허용하면 안 됩니다. 엔지니어의 분석이 맞습니다.**

편집에는 LOD 0 소스 데이터가 필요합니다. 메모리에 없다면:
- **디스크에서 온디맨드 로드** — 유일하게 실행 가능한 경로이나 IO 지연 발생.
- **모든 것을 메모리에 유지** — LOD 최적화의 목적을 무효화.
- **낮은 해상도로 편집** — LOD 0 소스가 나중에 로드될 때 아티팩트와 데이터 불일치 발생.

**다른 엔진의 방식:** 편집을 허용하기 전에 풀 해상도 청크를 메모리에 로드. Minecraft는 16x16x256 컬럼을 완전히 로드. Voxel Plugin for UE는 온디맨드로 LOD 0 데이터를 편집 버퍼에 로드.

**권장:** 플레이어가 편집 범위에 접근할 때, 근처 청크의 LOD 0 소스 데이터를 비동기적으로 프리로드. 편집 순간에 IO 비용을 한번에 부담하는 대신 시간에 걸쳐 분산. "편집 반경 내 LOD 0 프리페칭"으로 생각.

---

## 4. 수정된 LOD 0 청크 저장 후 LOD 업데이트 시

**문제:** LOD가 업데이트되고 밉맵 재생성이 필요하면, 수천 개의 소스 청크를 디스크에서 읽어야 할 수 있음.

**해법: 전체 재생성이 아닌 증분 밉맵 업데이트.**

단일 LOD 0 청크가 수정될 때, 부모 밉맵의 모든 소스를 다시 로드할 필요 없음. 밉맵 계층의 각 조상에 대한 *해당 청크의 기여분만* 업데이트하면 됩니다.

**기법 — 바텀업 증분 업데이트:**
1. LOD 0 청크 수정.
2. 수정된 청크 데이터를 LOD 1로 다운샘플링 (부모의 해당 사분면/팔분면만 교체).
3. 위로 전파: LOD 1 → LOD 2 → ... 등.
4. 각 단계에서 부모의 8개가 아닌 1개 팔분면만 처리.

이는 GPU 텍스처 스트리밍에서 2D 밉맵 업데이트가 작동하는 방식과 정확히 동일. 3D 복셀 밉맵에도 같은 원리 적용.

**IO 비용:** 수정된 청크 + 직접 조상 체인만. 깊이 N인 트리에서 N번의 읽기/쓰기, 수천 번이 아님.

**권장:** 증분 밉맵 전파 구현. 편집 시 전체 밉맵 재생성 절대 하지 않음 — 초기 월드 로드 시에만.

---

## 5. 밉맵 데이터를 어떻게 저장해야 하는가?

**밉맵을 소스가 아닌 캐시로 저장.**

- **소스 진실:** 디스크의 LOD 0 청크 파일.
- **밉맵 캐시:** LOD 레벨별 별도 파일, 폐기/재생성 가능한 것으로 취급.
- **로드 시:** 캐시된 밉맵이 존재하고 현재 월드 버전과 일치하는지 확인. 일치하면 사용. 아니면 소스에서 재생성.

**월드 크기와 스케일 의존성:**
밉맵 계층 구조(깊이, 분기 계수)는 월드 크기와 스케일에 의존. 데이터 영속화 후 이를 변경하면 모든 캐시된 밉맵이 무효화되어 재생성 필요.

**이것이 영속성 배포 전에 월드 크기와 스케일을 확정해야 하는 이유입니다.**

**저장 형식 권장:**
- 청크 좌표 + LOD 레벨로 명명된 LOD 레벨별 청크당 하나의 파일.
- RLE 압축 사용 — 복셀 데이터는 RLE로 매우 잘 압축됨 (실제로 2배~30배 압축률 보고).
- 월드 크기, 스케일, 밉맵 구조를 기록하는 매니페스트/버전 파일 유지. 파라미터 변경 시 캐시 무효화.

## 6. 밉맵 저장 스케줄링과 파이프라인

**확립된 패턴: 더티 플래깅 + 디바운스 Write-Behind.**

### 더티 플래깅
청크가 편집되면 더티로 표시. 밉맵 전파나 디스크 쓰기를 즉시 트리거하지 않음.

### 디바운싱
설정 가능한 윈도우(예: 새 편집 없이 200~500ms, 또는 최대 배치 윈도우 1~2초) 동안 편집을 축적. 그 후 모든 더티 청크를 한 번에 처리.

### 배치 밉맵 전파
디바운스가 발동되면:
1. 모든 더티 LOD 0 청크를 수집.
2. 각각에 대해 증분 밉맵 업데이트 수행 (#4 참조).
3. 중복 제거: 두 더티 청크가 밉맵 트리에서 부모를 공유하면, 부모는 한 번만 업데이트.
4. 수정된 청크 + 업데이트된 밉맵을 디스크에 기록.

### 우선순위 스케줄링
우선순위 큐 사용:
1. **최고 우선순위:** 리메싱이 필요한 플레이어에게 보이는 청크 (시각적 정확성).
2. **중간 우선순위:** 보이는 LOD 레벨의 밉맵 전파.
3. **최저 우선순위:** 디스크 영속화 (인메모리 상태보다 뒤처져도 됨).

**편집당 약 6개 밉맵 연산**은 디바운싱으로 충분히 감당 가능. 플레이어가 500ms 내에 20번 편집하면, 120번의 밉맵 연산이 아닌 하나의 배치로 처리.

### 인메모리 vs. 디스크 기반 밉맵 업데이트
- **부모 밉맵이 메모리에 있으면:** 제자리 업데이트. 빠름.
- **부모 밉맵이 디스크에만 있으면:** 읽기 → 패치 → 쓰기. 느리지만, 최근 접근한 밉맵을 캐시하면 드문 경우.
- **경험칙:** 보이는 LOD 레벨의 밉맵은 메모리에 유지. 화면 밖 LOD 레벨만 디스크 접근.

## 7. 월드 크기 및 스케일 조정

**영속성 배포 후 되돌릴 수 없기 때문에 가장 중요한 결정입니다.**

### 현재: 크기 14, 스케일 0.5. 제안: 스케일 0.25 (YYC와 동일).

**스케일 절반(0.5 → 0.25)의 의미:**
- 복셀 크기 절반 → 해상도 2배 → 볼륨당 복셀 수 8배.
- LOD 0 반경 절반 → LOD 전환이 플레이어에 더 가까이서 발생.
- 플레이어 이동 시 LOD 업데이트 빈도 증가.
- LOD 0 데이터의 메모리 압박 증가.

### LOD 업데이트 쓰래싱 완화 — 히스테리시스

빈번한 LOD 업데이트의 표준 해법은 **히스테리시스**(거리 임계값 또는 데드밴드):

- 플레이어가 경계를 넘을 때 LOD를 업데이트하지 않음. 대신 두 개의 임계값 사용: *업그레이드* 임계값(더 가까이)과 *다운그레이드* 임계값(더 멀리).
- 예시: 거리 50에서 LOD 0으로 업그레이드. 거리 60에서 LOD 1로 다운그레이드. 10단위 간격이 쓰래싱 방지.
- Voxel Plugin for UE는 이를 "Invoker Distance Threshold"로 부름 — LOD 옥트리는 플레이어가 이 임계값 이상 이동할 때만 재구축.

### 실질적 권장
- **스케일 0.25로 테스트하되**, 평가 전에 히스테리시스를 먼저 구현. 히스테리시스 없이는 어떤 작은 스케일이든 쓰래싱으로 보일 것.
- 편집 반경이 LOD 0 반경보다 작으면(당연히 그래야 함) 0.25 스케일의 LOD 0 반경은 수용 가능.
- LOD 0 반경을 스케일과 독립적으로 설정 가능하게 만드는 것을 고려 — 시각적 LOD와 편집 범위 LOD를 분리.

### 고착 리스크
- 최종 스케일로 배포. 나중에 바꿀 계획 세우지 말 것.
- 확실하지 않으면, **작은 스케일(0.25)**과 큰 월드 크기로 시작. 공간을 낭비하는 것이 영속화된 월드를 리사이즈하는 것보다 쉬움.

## 8. LOD 전파 로직 변경

**필요한 변경 (#2와 #7에 의해):**

### 하드 제약: 이웃 간 최대 LOD 차이 1
선택 사항이 아님 — 다음을 위해 필수:
- Transvoxel 메싱 (2배 해상도 경계만 처리).
- 시각적 정확성 (크랙/이음새 없음).
- 편집 일관성 (어떤 청크가 LOD 0인지 파악).

### 청크 처리 수
스케일 0.25에서 LOD 0 볼륨은 작지만, 월드의 총 청크 수는 더 많음. 전파 로직이 업데이트당 더 많은 청크를 처리.

**완화책:**
- **히스테리시스** (#7 참조) — 업데이트 빈도 감소.
- **증분 전파** — 플레이어가 임계값을 넘을 때만 재전파하고, LOD가 실제로 변경되는 청크만 처리.
- **예산 시스템** — 프레임당 LOD 전환 수를 제한. 작업을 여러 프레임에 분산. Voxel Tools(Godot)가 이 접근법 사용: 프레임레이트 유지를 위해 프레임당 청크 로드/메시 수 제한.

### 브라우저 예산
WebGL/브라우저의 합리적 목표:
- 프레임당 최대 4~8개 청크 메시 리빌드.
- 프레임당 최대 2~4개 청크 디스크 로드 (Web Worker의 OPFS 경유).
- LOD 전파 자체는 저렴(트리 순회) — 비용은 결과적인 메시/로드 작업에서 발생.

## 브라우저 특화: OPFS를 통한 스토리지

**IndexedDB는 답이 아닙니다.** Chrome에서 대용량 바이너리 데이터에 심각한 성능 문제가 있습니다.

**OPFS (Origin Private File System)가 올바른 선택:**
- IndexedDB 대비 바이너리 데이터에서 3~4배 빠름.
- `createSyncAccessHandle()`을 통한 동기 읽기/쓰기 지원 (Web Worker에서만).
- 정확히 이 사용 사례를 위해 설계: 대용량 바이너리 블롭, 높은 빈도의 접근.

### 아키텍처
- **메인 스레드:** 게임 루프, 렌더링, 플레이어 입력.
- **Web Worker 1:** 청크 직렬화/역직렬화, OPFS 읽기/쓰기.
- **Web Worker 2 (선택):** 밉맵 계산, 메시 생성.

### OPFS 파일 레이아웃
```
/world/
  manifest.json          # 월드 크기, 스케일, 버전
  /chunks/
    /lod0/
      chunk_x_y_z.bin    # RLE 압축된 LOD 0 소스
    /lod1/
      chunk_x_y_z.bin    # 캐시된 밉맵 (재생성 가능)
    /lod2/
      ...
```

### 핵심 제약
`createSyncAccessHandle()`은 메인 스레드가 아닌 Web Worker에서만 동작. 모든 디스크 IO는 Worker로 오프로드해야 함. 이는 실제로 좋은 아키텍처 — 비동기 IO를 강제하고 프레임 드롭 방지.

## 권장 구현 순서

1. **월드 크기와 스케일 확정.** 이것이 나머지 모든 것을 블록.
2. **편집 반경 내 플레이어 위치 기반 LOD 0 강제 구현.** 이웃 간 최대 LOD 차이 1 강제.
3. **LOD 0 청크 직렬화를 OPFS로** (Web Worker). 개별 청크 저장/로드.
4. **더티 플래깅 + 디바운스 저장 파이프라인.**
5. **증분 밉맵 전파** (바텀업, 레벨당 하나의 팔분면).
6. **밉맵 캐싱을 OPFS로.**
7. **프레임당 청크 처리 예산 시스템.**
8. **LOD 업데이트 거리 임계값 히스테리시스.**
